# TASK.md

## Цель

Реализовать полноценное веб-приложение для управления задачами на Django 6.0.3 и Python 3.14 с двумя интерфейсами:

1. HTML-веб-интерфейс на Django Templates.
2. Полноценный REST API на Django REST Framework.

Проект должен быть выполнен как качественное тестовое задание уровня production-like: с чистой архитектурой, типизацией, линтерами, тестами, документацией и понятными инструкциями по запуску.

Основной сценарий локальной разработки и запуска должен быть построен вокруг `uv`. Под это нужно подготовить настройки проекта и документацию. В `README.md` обязательно нужно добавить отдельный раздел для пользователей, у которых `uv` не установлен: либо как установить `uv`, либо как запустить проект через стандартный `venv + pip` без `uv`.

## Главные правила разработки

### Обязательные требования

1. Строго соблюдать PEP 8.
2. Во всех Python-файлах использовать `line-length = 121`.
3. Все функции, классы, методы и модули должны иметь докстроки на русском языке.
4. Все докстроки писать только в тройных двойных кавычках.
5. Все обычные строки в Python-коде, логах, константах, сообщениях и т. п. писать в одинарных кавычках.
6. Весь код должен быть совместим с Python 3.14.
7. Использовать Django 6.0.3.
8. Использовать Django REST Framework.
9. База данных: SQLite.
10. Веб-интерфейс реализовать через Django Templates.
11. Интерфейс должен быть аккуратным, современным, спокойным и удобным.
12. Подобрать ненавязчивую и не банальную палитру. Предпочтение:
    - фон: мягкий светлый серо-зелёный или серо-бежевый,
    - карточки: почти белые,
    - акцент: приглушённый сине-зелёный / тёмный шалфейный / пыльный индиго,
    - статусные цвета: мягкие, без кислотности.
13. Использовать актуальные стабильные версии инструментов, совместимые с Python 3.14 и Django 6.0.3:
    - `ruff`
    - `mypy`
    - `pytest`
    - `pytest-asyncio`
    - `pytest-django`
14. Все настройки инструментов хранить в `pyproject.toml`.
15. Локальный запуск и основная работа с проектом должны быть ориентированы на `uv`:
    - зависимости устанавливаются через `uv sync`,
    - команды запускаются через `uv run ...`,
    - при необходимости должен быть зафиксирован `uv.lock`.
16. Архитектура должна быть обоснованной и чистой.
17. Основная логика должна быть покрыта тестами.
18. Использовать `conftest.py`.
19. Тесты писать асинхронными там, где это применимо.
20. Указать в настройках pytest:
    - `asyncio_mode = 'auto'`
    - markers:
      - `unstable: Нестабильные тесты.`
      - `integration: Интеграционные тесты.`
      - `module: Юнит-тесты.`
21. Автосгенерированные файлы уровня миграций не нужно искусственно перегружать лишними докстроками или ручными правками, если это ухудшает поддержку. Для прикладного кода требования по докстрокам обязательны.

## Важное замечание по асинхронности

Нужно использовать асинхронные возможности Django там, где это оправдано:

- async views для части представлений,
- async API views там, где это не усложняет код,
- асинхронные тесты,
- учесть совместимость Django 6.0.3 и DRF.

При этом:

- не делать псевдо-асинхронность ради галочки,
- не ломать читаемость,
- если какой-то участок логичнее оставить синхронным, это допустимо,
- формулировку 'мультиворкеры для максимальной скорости' трактовать как требование подготовить проект к запуску под production ASGI-сервером, а не встраивать это в код приложения.

В документации по запуску указать:

- рекомендуемый вариант локального запуска через `uv`,
- рекомендуемый production-подход через ASGI-сервер с несколькими воркерами,
- отдельный fallback-раздел для тех, кто запускает проект без `uv`.

## Что должно быть реализовано

### Основной функционал

Реализовать систему управления задачами со следующими возможностями:

1. Создание задачи.
2. Редактирование задачи.
3. Удаление задачи.
4. Просмотр списка задач.
5. Просмотр детальной страницы задачи.
6. Назначение задачи пользователю.
7. Отметка задачи как выполненной.
8. Повторное открытие задачи.
9. Комментирование задачи.

### Дополнительно, обязательно реализовать

Хотя в исходном задании часть пунктов названа опциональной, в этом проекте нужно реализовать их тоже:

1. Авторизация пользователей по JWT для API.
2. Swagger / OpenAPI-документация.
3. Обычную аутентификацию для HTML-интерфейса через Django auth.
4. Разграничение доступа:
   - только авторизованные пользователи работают с системой,
   - пользователь видит только те задачи, где он является автором или назначенным исполнителем,
   - пользователь может создавать задачи,
   - автор задачи и назначенный исполнитель могут видеть и редактировать связанную задачу,
   - автор задачи может назначить исполнителем любого существующего пользователя системы,
   - комментировать задачу могут только авторизованные пользователи, у которых есть доступ к задаче,
   - удаление задачи - только автор задачи.

Если в ходе реализации потребуется минимально уточнить права, использовать именно эти правила и не придумывать альтернативные трактовки.

## Модель предметной области

Нужно реализовать минимум следующие сущности.

### Пользователь

Использовать стандартную модель пользователя Django через `get_user_model()`.
Кастомную модель создавать только если это действительно необходимо. По умолчанию - не создавать.

### Задача

Поля:

1. `id`
2. `title` - короткий заголовок задачи
3. `description` - подробное описание задачи
4. `status` - статус задачи
5. `author` - создатель задачи
6. `assignee` - назначенный пользователь, nullable
7. `is_completed` - признак выполнения
8. `created_at`
9. `updated_at`
10. `completed_at` - дата и время завершения, nullable

### Комментарий

Поля:

1. `id`
2. `task` - задача
3. `author` - автор комментария
4. `text`
5. `created_at`
6. `updated_at`

### Статус задачи

Реализовать через `TextChoices`, минимум такие значения:

- `new`
- `in_progress`
- `done`

Правила:

- если задача помечается как выполненная, то `is_completed = True`,
- при завершении автоматически выставлять `completed_at`,
- если задача переведена обратно в незавершённую, очищать `completed_at`,
- статус `done` должен быть синхронизирован с признаком выполнения,
- нельзя допускать противоречивое состояние, когда `status = done`, а `is_completed = False`, или наоборот.

Эту согласованность лучше инкапсулировать в доменной логике, а не размазывать по всему проекту.

## Требования к API

Нужно реализовать полностью функциональный REST API.

### Эндпоинты

Минимально нужны такие маршруты:

#### Аутентификация

- `POST /api/auth/jwt/create/`
- `POST /api/auth/jwt/refresh/`
- `POST /api/auth/jwt/verify/`

Можно использовать `djangorestframework-simplejwt`.

#### Пользователи

- `GET /api/users/me/`

#### Задачи

- `GET /api/tasks/`
- `POST /api/tasks/`
- `GET /api/tasks/{id}/`
- `PATCH /api/tasks/{id}/`
- `DELETE /api/tasks/{id}/`
- `POST /api/tasks/{id}/complete/`
- `POST /api/tasks/{id}/reopen/`

#### Комментарии

- `GET /api/tasks/{id}/comments/`
- `POST /api/tasks/{id}/comments/`

Допустимо сделать и отдельный endpoint комментариев, но доступ к ним должен быть привязан к задаче.

### Требования к API-поведению

1. Ответы должны быть предсказуемыми.
2. Ошибки должны возвращать корректные HTTP status codes.
3. Валидация должна быть понятной.
4. Нельзя позволять назначить задачу несуществующему пользователю.
5. Нельзя позволять пустой заголовок задачи.
6. Нельзя позволять пустой текст комментария.
7. При попытке доступа без прав должен быть `403`.
8. При отсутствии сущности - `404`.
9. JWT должен работать корректно.
10. Список задач в API должен фильтроваться по доступным пользователю сущностям, а не отдавать чужие задачи без необходимости.
11. При назначении исполнителя автор задачи может выбрать любого существующего пользователя системы.
